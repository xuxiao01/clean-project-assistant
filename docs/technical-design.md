# 清洁项目助手 Demo 技术方案说明

## 1. 项目概述

### 1.1 项目背景
本项目旨在实现一个“清洁项目助手”Demo。用户通过一个简单页面输入问题，系统基于预先提供的项目文档内容进行检索，并结合大模型生成回答，用于辅助查询项目相关信息。

该项目当前定位为一个本地演示 Demo，后续可作为 BFF（Backend For Frontend）能力嵌入到其他业务系统页面中使用。

### 1.2 项目目标
实现一个最小可运行的项目问答原型，具备以下能力：

- 提供一个简单的 Web 页面用于用户输入问题
- 能够读取项目相关文档
- 能够对文档内容进行拆分与检索
- 能够将检索结果作为上下文传递给大模型
- 能够返回大模型生成的项目问答结果

### 1.3 项目定位
本项目当前阶段为快速交付的单人 Demo 项目，重点目标是：

- 优先跑通完整链路
- 优先保证开发速度
- 优先贴合现有技术背景
- 优先降低开发和学习成本

不追求当前阶段的复杂工程化能力，如多角色权限、复杂知识库管理、分布式部署、长期稳定运行等。

---

## 2. 需求分析

### 2.1 用户角色
当前用户为开发者本人，仅用于本地演示和方案验证。

### 2.2 核心功能
系统需要具备以下核心功能：

1. 简单问答页面
   - 提供输入框
   - 提供提交按钮
   - 展示 AI 返回结果

2. 项目文档处理
   - 支持读取项目文档内容
   - 支持将文档内容拆分为多个文本片段（chunk）

3. 文档检索
   - 根据用户问题，从文档片段中检索相关内容
   - 将相关片段作为上下文提供给大模型

4. 大模型问答
   - 调用外部大模型 API
   - 基于检索结果生成回答

### 2.3 非功能需求
当前阶段无特殊非功能要求，但需满足以下基本目标：

- 本地可运行
- 易于快速开发
- 结构清晰，便于后续调整
- 尽量减少外部依赖和部署成本

---

## 3. 技术选型结论

## 3.1 最终方案
本项目采用以下技术方案：

- 前端：HTML / 可选最小化 Vue3 页面
- 后端：Node.js + TypeScript + Express
- 文档格式：优先使用 Markdown（.md）或纯文本（.txt）
- 文档拆分：基于规则的文本切片
- 向量存储：内存向量库 / 本地 JSON 持久化
- 检索方式：基于向量相似度的 Top-K 检索
- 大模型服务：Gemini API
- 部署方式：本地运行演示

### 3.2 选型原则
本方案遵循以下原则：

- 优先选择本人熟悉的技术栈
- 优先满足快速交付 Demo 的要求
- 优先降低学习成本和调试成本
- 避免过度工程化
- 保留后续扩展空间

---

## 4. 技术栈说明

## 4.1 前端技术
### 选型
- HTML + JavaScript（最小实现）

### 选型原因
当前页面需求极简，仅需要输入问题和展示回答，因此无需引入复杂前端架构。若希望与后续已有前端项目更自然衔接，可采用 Vue3 + TypeScript 实现一个最小页面组件。

### 当前建议
为了优先跑通链路，建议先采用最简单页面实现。若后续需要接入现有系统，再封装为 Vue3 组件。

---

## 4.2 后端技术
### 选型
- Node.js
- TypeScript
- Express

### 选型原因
1. 与个人技术背景最匹配  
   开发者当前主要技术栈为 Vue3 + TypeScript，因此 Node.js + TypeScript 可以实现前后端语言统一，降低理解和开发成本。

2. 适合快速开发  
   Express 生态成熟，适合快速搭建简单接口服务。

3. 适合 AI 辅助开发  
   对于 TypeScript 项目，Cursor 可以更好地进行类型推断、模块补全、文件间跳转和代码生成，开发体验更优。

---

## 4.3 文档处理技术
### 选型
- 第一阶段仅支持 `.md` / `.txt`
- 后续可扩展支持 `.docx`
- 暂不优先支持 `.pdf`

### 选型原因
1. Markdown / 纯文本最容易解析  
   可以避免 PDF 排版、页眉页脚、扫描版文本抽取等问题。

2. 更适合先验证 RAG 链路  
   当前项目重点不是文档解析能力，而是“文档检索 + 大模型回答”主链路。

3. 便于 AI 工具协助开发  
   使用结构清晰的 Markdown 文档，更便于 Cursor 和开发者理解文档内容与调试检索结果。

---

## 4.4 向量检索方案
### 选型
- 内存向量库
- 本地 JSON 持久化（可选）

### 选型原因
1. 当前数据规模小  
   Demo 阶段文档量不会很大，没有必要引入独立向量数据库。

2. 降低环境复杂度  
   不需要安装 Qdrant、Milvus、PostgreSQL + pgvector 等额外服务。

3. 有助于快速调试  
   检索逻辑、文档切片和向量结果更容易直接查看和排查问题。

### 当前限制
- 不适合大规模文档场景
- 数据持久化能力较弱
- 后续正式化时可能需要升级为独立向量数据库

---

## 4.5 大模型服务选型
### 选型
- Gemini API

### 选型原因
1. Demo 阶段优先考虑低成本甚至免费方案
2. 云端模型接入简单，避免本地模型部署成本
3. 响应速度和效果通常比本地轻量模型更稳定
4. 对单人快速开发更友好

### 不选本地模型的原因
虽然本地模型方案具有成本低和本地化优势，但当前阶段不优先考虑，原因如下：

- 本地模型部署与调试复杂度更高
- 对机器性能有要求
- 容易把时间消耗在模型和环境问题上
- 不利于短时间内稳定做出演示效果

---

## 5. 整体架构设计

## 5.1 架构说明
本项目采用轻量化前后端分离结构。

### 架构流程
1. 开发者准备项目知识文档（Markdown）
2. 后端读取文档内容
3. 后端将文档拆分为多个 chunk
4. 后端为每个 chunk 生成向量并保存到内存中
5. 用户在页面输入问题
6. 后端对问题生成向量
7. 后端从已存储的 chunk 中检索最相关的 Top-K 文本片段
8. 后端构造 Prompt，将问题与上下文一起发送给大模型
9. 大模型返回回答
10. 前端展示回答结果

---

## 5.2 架构分层
建议将系统拆分为以下模块：

### 1. 文档加载层
负责读取本地文档内容。

### 2. 文档切分层
负责将长文档拆分为适合检索的文本片段。

### 3. 向量生成层
负责将文本片段和用户问题转换为向量表示。

### 4. 检索层
负责根据用户问题向量检索最相似的文档片段。

### 5. Prompt 构建层
负责将问题和上下文组装为发给大模型的 Prompt。

### 6. 模型调用层
负责调用 Gemini API 获取回答。

### 7. 接口层
负责对前端暴露 HTTP 接口。
